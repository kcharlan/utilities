# Agent Pipeline — System Rules

You are operating in a pipelined multi-agent work system. Your specific role
(Planner or Worker) is defined in the prompt file for your directory.

## Directory Structure

```
work/
  SYSTEM.md              ← you are here (shared rules)
  planning/
    PLANNER.md           ← planner agent instructions
    intake/              ← raw work items (one .md file per item)
    claimed/             ← planner grabbed it, writing plan
    staging/             ← plan complete, awaiting dependency resolution
    review/              ← plan needs human input before it's ready
  execution/
    RESOLVER.md          ← dependency resolver agent instructions
    WORKER.md            ← worker agent instructions
    ready/               ← dependency-resolved, queued for execution
    workers/             ← per-worker slots (parallel mode)
      0/                 ← worker slot 0
        *.plan.md        ← current plan being executed
        *.status         ← status sidecar
        *.log            ← worker output
      1/                 ← worker slot 1
      ...
    active/              ← (legacy) single-worker execution
    done/                ← completed successfully
    blocked/             ← worker hit a wall, needs escalation
  DASHBOARD.md           ← generated by orchestrator, read-only
```

## File location is state

Plans move through directories. The full path is:

```
intake/ → claimed/ → staging/ → [resolver] → ready/ → active/ → done/
                         ↓                                         ↓
                      review/                                  blocked/
```

**Planning phase** (planner-owned):
`intake/ → claimed/ → staging/` (or `review/` if questions exist)

**Dependency resolution** (resolver-owned):
`staging/ → ready/` — the resolver reads ALL plans in `staging/` as a batch,
analyzes cross-plan dependencies and file overlaps, sets DEPENDS_ON (hard
ordering), ANTI_AFFINITY (concurrent execution prevention), and EXEC_ORDER,
then moves resolved plans to `ready/`.

**Execution phase** (orchestrator-owned):
`ready/ → workers/<slot>/ → done/ | blocked/`

In parallel mode, the orchestrator dispatches individual plans to worker slots,
each in its own git worktree. Plans run one per worktree. On completion, the
orchestrator merges the worktree branch, tears down the worktree, then moves
the plan to `done/` or `blocked/`. The orchestrator enforces two constraints:
hard dependencies (DEPENDS_ON — plan must wait for another to complete) and
anti-affinity (ANTI_AFFINITY — plans sharing files cannot run concurrently but
have no ordering requirement).

**Review detour** (human-owned):
When the planner has questions that would materially affect the implementation,
the plan goes to `review/` instead of `staging/`. Two resolution paths exist:

**Path A — Quick turnaround** (you're busy, keep the pipeline moving):
The human adds answers/directives to the top of the plan and drops it back
into `intake/`. The next planner pass picks it up, revises the plan using the
answers, and routes the updated plan to `staging/` (or back to `review/` if
new questions emerged).
`review/ → intake/ (with answers) → claimed/ → staging/`

**Path B — Deep dive** (complex problem, needs interactive discussion):
The human opens an interactive session with an agent to work through the
questions, revises the plan in place, and moves it directly to `staging/`
(for the resolver to pick up) or to `ready/` (if it has no dependencies).
`review/ → (human + agent) → staging/ or ready/`

If a worker can't complete a plan, it lands in `blocked/`:
`active/ → blocked/ → (human resolves) → ready/ (retry)`

**Who moves what:**
- Planners: `intake/` → `claimed/`, output to `staging/` or `review/`,
  then delete the original from `claimed/`
- Resolver: `staging/` → `ready/`
- Orchestrator: `ready/` → `workers/<slot>/` → `done/` or `blocked/`
  (also manages worktree creation, merging, and cleanup)
- Fixer agent: invoked by orchestrator on failure. Operates in the existing
  worktree (plan failure) or main repo (test suite failure). Commits fixes
  in-place. Does not move files between directories.
- Human: `review/` → `intake/` or `staging/` or `ready/`;
  `blocked/` → `ready/`

## Status sidecar format

When a worker finishes (success or failure), it writes a `.status` file
alongside the plan file — in `execution/workers/<slot>/` (parallel mode)
or `execution/active/` (legacy single-worker mode). The filename matches the
plan but with a `.status` extension instead of `.plan.md`.

Example: plan `001_add_export_api.plan.md` gets sidecar
`001_add_export_api.status`.

Contents:

```
STATUS: done | blocked | error
COMMITS: <comma-separated SHAs, or "none">
TESTS_RAN: targeted | full
TEST_RESULT: pass | fail
BLOCKED_REASON: <one line, only if STATUS is blocked or error>
NOTES: <optional one line>
```

## Operator Actions and Release Notes

Every plan includes a `## Operator Actions` section that documents post-deployment
requirements (infrastructure provisioning, data migrations, configuration changes,
breaking changes, rollback notes). Planners write it, workers preserve it.

At session end, the orchestrator runs `generate_release_notes.sh`, which
aggregates these sections from all completed plans into `work/RELEASE_NOTES.md`.
This gives the operator a single document listing everything they need to do
before and after deploying the new code.

## Rules for all agents

1. Never edit `DASHBOARD.md` — it is generated by the orchestrator.
2. Each agent role owns specific directory transitions (see "Who moves what"
   above). Do not move files outside your role's boundaries.
3. Planners: `intake/` → `claimed/`, output to `staging/` or `review/`.
4. Resolver: `staging/` → `ready/` (in `execution/`).
5. Workers: write `.status` files in `active/` when finished.
6. If you hit a wall after two attempts at the same problem, stop. Write
   STATUS: blocked with a clear explanation and exit.
7. Use `.venv/bin/pytest` for all Python tests, never system pytest.
8. Follow all conventions in the repo-root `CLAUDE.md`.
9. **Branch safety:** The orchestrator will refuse to run if the current
   branch is `main`. All pipeline work happens on feature/development
   branches. Never commit pipeline output directly to `main`.
10. **Auto-fix:** When a worker fails or the full test suite breaks, the
    orchestrator automatically launches a fixer agent (up to 2 attempts)
    before escalating to blocked/halt. Plans only reach `blocked/` after
    auto-fix has been exhausted.
11. **E2E tests are not optional.** If a plan touches UI-visible behavior
   (frontend code, API endpoints that serve the UI, auth flows, navigation),
   the planner must specify an e2e test and the worker must write the
   Playwright spec. Do NOT punt to "manual verification" or "visual
   spot-check." Existing specs in `tests/e2e/` show the patterns.

## The review/ flow

When a planner encounters questions that would materially affect the
implementation — ambiguous requirements, multiple valid approaches, missing
context, or anything where guessing wrong would waste the worker's time — it
writes the plan to `review/` instead of `staging/`.

The plan file in `review/` must include a `## Questions for Review` section
at the top (before the implementation steps) listing each open question with
enough context for the human to make a decision. The rest of the plan should
be as complete as possible given what IS known — draft the implementation
assuming the planner's best-guess answer, and note which steps would change
depending on the answer.

### Resolution path A: Quick turnaround (refeed into pipeline)

The human adds answers or directives at the top of the plan file (above
`## Questions for Review`) and moves it back to `intake/`:

`mv work/planning/review/<plan>.plan.md work/planning/intake/`

The next planner picks it up like any other intake item. When a planner
claims a `.plan.md` file (as opposed to a plain `.md` intake item), it
recognizes this as a **revision pass**: read the human's answers, revise
the plan accordingly, remove the resolved `## Questions for Review`
section, and route to `staging/` — or back to `review/` if new questions
emerged from the answers.

### Resolution path B: Deep dive (interactive session)

The human opens an interactive agent session to work through the questions:

`cc-opus -p "Read work/planning/review/<plan_file>. Here are my answers: ..."`

Or loads the file and discusses interactively. After the plan is revised:
1. The `## Questions for Review` section is removed (or marked resolved).
2. The human moves the finalized plan to `staging/` (normal — goes through
   dependency resolution) or directly to `execution/ready/` (if you're
   certain it has no cross-plan dependencies):
   `mv work/planning/review/<plan>.plan.md work/planning/staging/`
